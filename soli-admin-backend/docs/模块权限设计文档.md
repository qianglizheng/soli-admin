# 模块权限设计文档

## 1. 文档说明

本文档基于当前仓库代码现状整理，目标是把模块权限平台相关设计说明为一份可直接用于研发、联调和后续扩展的设计文档。文档覆盖以下 5 个模块：

1. 模块管理
2. 岗位管理
3. 功能授权
4. 字段标题
5. 状态权限

代码基线范围：

- 后端：`soli-admin-backend/soli-admin-framework/soli-admin-system-api`
- 后端：`soli-admin-backend/soli-admin-framework/soli-admin-system-core`
- 后端：`soli-admin-backend/soli-admin-framework/soli-admin-system-web`
- 前端：`soli-admin-frontend/src/views/system`
- 初始化脚本：`soli-admin-backend/sql/init_Table.sql`、`soli-admin-backend/sql/init_Data.sql`

本文档强调“现状设计”，即以当前已经落地的控制器、Service、Mapper、前端页面和 SQL 为准，不把未实现的能力写成已实现。

## 2. 总体设计概览

### 2.1 设计目标

当前模块权限平台的核心目标是：

- 用“模块元数据”统一驱动后台导航、页面结构、字段定义、按钮定义和状态定义。
- 用“岗位”作为业务授权主体，替代传统以角色为中心的单据内权限控制方式。
- 把“字段标题”从技术字段定义中拆开，单独维护展示文案。
- 把“状态权限”从岗位基线权限中拆开，只做状态维度的附加收紧。
- 让系统管理页面与控制器鉴权都基于模块权限平台运行。

### 2.2 三层落点

当前工程按典型三层结构落地：

- `soli-admin-system-api`
  - 定义 DTO、Query、CreateRequest、ModifyRequest、Service 接口
- `soli-admin-system-core`
  - 定义 Entity、Model、Converter、Mapper、Mapper.xml、ServiceImpl
- `soli-admin-system-web`
  - 定义 Controller，对外暴露 REST 接口

前端按页面拆分：

- `src/views/system/module-center`
- `src/views/system/post-manage`
- `src/views/system/function-auth`
- `src/views/system/module-title`
- `src/views/system/state-auth`

### 2.3 核心权限模型

当前代码采用四层解耦模型：

1. `模块元数据层`
   - 定义模块、Tab、字段、按钮、状态、流转。
2. `岗位基线权限层`
   - 定义某岗位对某模块的可见性、字段权限、按钮权限。
3. `字段标题层`
   - 定义字段展示标题、占位提示、帮助文案。
4. `状态收紧层`
   - 定义某模块在某状态下，对字段和按钮做进一步收紧。

最终权限计算规则为：

- 字段：`finalFieldLevel = min(岗位字段权限, 状态字段限制)`
- 按钮：`finalButtonLevel = min(岗位按钮权限, 状态按钮限制)`

其中：

- 字段权限：`0 = 隐藏`，`1 = 只读`，`2 = 可写`
- 按钮权限：`0 = 隐藏`，`1 = 禁用`，`2 = 可用`
- 状态字段限制：`0 = 隐藏`，`1 = 只读`，`2 = 不收紧`
- 状态按钮限制：`0 = 隐藏`，`1 = 禁用`，`2 = 不收紧`

### 2.4 当前系统边界

当前代码已经落地的范围：

- 系统导航由模块树动态生成。
- Controller 权限校验由模块可见性和按钮可用性决定。
- 模块管理已支持模块、Tab、字段、按钮维护。
- 岗位管理已支持组织树、岗位树、岗位员工绑定。
- 功能授权已支持岗位基线权限维护。
- 字段标题已支持字段展示文案维护。
- 状态权限已支持状态维度的字段/按钮限制维护。

当前代码尚未独立落地的范围：

- 没有单独的“运行时权限上下文服务”对业务单据进行完整装配。
- `GET /sys/module/{id}/context-preview` 只做模块结构预览，不是业务运行时上下文。
- 模块状态与流转虽然有表结构、种子数据和详情返回，但当前没有独立的状态/流转 CRUD 控制器接口。

## 3. 领域关系设计

### 3.1 关系总览

```text
sys_org_unit
    └─ sys_org_post
         └─ sys_user_org_post
              └─ sys_user

sys_module
    ├─ sys_module_tab
    │    └─ sys_module_field
    │         └─ sys_module_field_title
    ├─ sys_module_button
    ├─ sys_module_state
    └─ sys_module_transition

sys_org_post + sys_module
    └─ sys_org_post_module_auth

sys_org_post + sys_module_field
    └─ sys_org_post_field_auth

sys_org_post + sys_module_button
    └─ sys_org_post_button_auth

sys_module + state_code + sys_module_field
    └─ sys_module_state_field_auth

sys_module + state_code + sys_module_button
    └─ sys_module_state_button_auth
```

### 3.2 模块之间的关系

#### 岗位管理

- 负责维护“授权主体”。
- 给功能授权提供岗位树和岗位详情。
- 给控制器鉴权、导航鉴权提供用户所属岗位映射。

#### 模块管理

- 是其他四个模块的主数据来源。
- 给功能授权提供模块树、Tab、字段、按钮。
- 给字段标题提供字段清单。
- 给状态权限提供状态、流转、字段、按钮。

#### 功能授权

- 建立“岗位 + 模块”的基线授权关系。
- 决定模块是否可见、导航是否显示、字段是否可写、按钮是否可用。

#### 字段标题

- 在不改动技术字段定义的前提下，覆盖字段展示文案。
- 通过 `sys_module_field_title` 与 `sys_module_field` 解耦。

#### 状态权限

- 对功能授权的结果做状态维度补充限制。
- 只允许收紧，不允许放大。

## 4. 表结构设计

## 4.1 组织与岗位域

### 4.1.1 `sys_org_unit`

作用：维护组织节点，当前岗位树中的组织部分来自该表。

关键字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 组织 ID |
| `parent_id` | 父组织 ID |
| `ancestors` | 祖级路径 |
| `org_code` | 组织编码，唯一 |
| `org_name` | 组织名称 |
| `org_type` | 组织类型，当前种子数据包含 `HEADQUARTERS`、`BRANCH` |
| `leader_user_id` | 负责人用户 ID |
| `status` | 状态 |

关键索引：

- `uk_sys_org_unit_code`
- `idx_sys_org_unit_parent`
- `idx_sys_org_unit_type_status`

说明：

- 当前 Service 中新增组织单元时，固定落为 `BRANCH`。
- 组织单元更多承担组织树作用，不直接存放权限数据。

### 4.1.2 `sys_org_post`

作用：维护岗位树，是业务授权的主体表。

关键字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 岗位 ID |
| `org_unit_id` | 所属组织 ID |
| `parent_post_id` | 上级岗位 ID |
| `ancestors` | 岗位祖级路径 |
| `post_code` | 岗位编码，同组织内唯一 |
| `post_name` | 岗位名称 |
| `post_type` | 岗位类型 |
| `manager_user_id` | 岗位负责人 |
| `status` | 状态 |

关键索引：

- `uk_sys_org_post_unit_code`
- `idx_sys_org_post_parent`
- `idx_sys_org_post_unit_status`

说明：

- 岗位不允许跨组织移动。
- 删除岗位前必须确保没有子岗位、没有绑定员工。

### 4.1.3 `sys_user_org_post`

作用：维护用户与岗位的映射关系，是后续导航与按钮鉴权的入口数据。

关键字段：

| 字段 | 说明 |
| --- | --- |
| `user_id` | 用户 ID |
| `org_post_id` | 岗位 ID |
| `primary_flag` | 是否主岗位 |
| `status` | 状态 |

关键索引：

- `uk_sys_user_org_post`
- `idx_sys_user_org_post_post`
- `idx_sys_user_org_post_user`

说明：

- 一个用户可以绑定多个岗位。
- 当前导航和按钮鉴权逻辑按用户的所有有效岗位求并集。

## 4.2 模块元数据域

### 4.2.1 `sys_module`

作用：模块主表，定义系统导航节点、页面节点和业务模块节点。

关键字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 模块 ID |
| `parent_id` | 父模块 ID |
| `ancestors` | 模块树路径 |
| `module_code` | 模块编码，唯一 |
| `module_name` | 模块名称 |
| `module_type` | `CATALOG` / `PAGE` / `BILL` |
| `route_path` | 前端路由片段 |
| `component_path` | 前端组件路径 |
| `icon` | 图标 |
| `nav_visible` | 是否在导航展示 |
| `stateful_flag` | 是否状态型模块 |
| `state_field_code` | 状态字段编码 |
| `context_version` | 上下文版本 |
| `status` | 状态 |

关键索引：

- `uk_sys_module_code`
- `idx_sys_module_parent`
- `idx_sys_module_type_status`

说明：

- `context_version` 是整个模块结构及权限变更的统一版本号。
- 模块新增、修改、字段标题变更、岗位权限变更、状态权限变更都会推动版本递增。

### 4.2.2 `sys_module_tab`

作用：定义模块下的单头/明细 Tab。

关键字段：

| 字段 | 说明 |
| --- | --- |
| `module_id` | 所属模块 |
| `tab_scope` | `HEADER` / `DETAIL` |
| `tab_code` | Tab 编码 |
| `tab_name` | Tab 名称 |
| `sort` | 排序 |

关键索引：

- `uk_sys_module_tab`
- `idx_sys_module_tab_scope`

说明：

- 同一模块、同一作用域下 Tab 编码唯一。
- 字段标题与功能授权不再复制 Tab 表，统一复用该表。

### 4.2.3 `sys_module_field`

作用：定义技术字段，是字段权限和字段标题的主数据来源。

关键字段：

| 字段 | 说明 |
| --- | --- |
| `module_id` | 模块 ID |
| `tab_id` | Tab ID |
| `field_scope` | 作用域 |
| `field_code` | 字段编码 |
| `default_title` | 默认标题 |
| `component_type` | 组件类型 |
| `data_path` | 数据路径 |
| `value_type` | 值类型 |
| `required_flag` | 是否必填 |
| `sort` | 排序 |

关键索引：

- `uk_sys_module_field`
- `idx_sys_module_field_tab`
- `idx_sys_module_field_scope`

说明：

- 当前查询字段时，会左连 `sys_module_field_title`，直接回填 `display_title`、`placeholder`、`help_text`。
- 新增字段时，系统会自动初始化一条默认标题记录，并自动给 admin 岗位补齐可写权限。

### 4.2.4 `sys_module_field_title`

作用：维护字段展示层文案，不改变技术字段结构。

关键字段：

| 字段 | 说明 |
| --- | --- |
| `field_id` | 字段 ID |
| `locale` | 语言区域，当前固定 `zh_CN` |
| `display_title` | 展示标题 |
| `placeholder` | 占位提示 |
| `help_text` | 帮助文案 |

关键索引：

- `uk_sys_module_field_title`
- `idx_sys_module_field_title_status`

说明：

- 当前保存逻辑中，如果 `display_title` 和 `default_title` 一致，则实际落库为 `null`，表示回退默认标题。

### 4.2.5 `sys_module_button`

作用：定义模块按钮，是按钮权限的主数据来源。

关键字段：

| 字段 | 说明 |
| --- | --- |
| `module_id` | 模块 ID |
| `button_code` | 按钮编码 |
| `default_title` | 默认标题 |
| `button_area` | 按钮区域 |
| `sort` | 排序 |

关键索引：

- `uk_sys_module_button`
- `idx_sys_module_button_area`

说明：

- 当前前端按钮区域分为 `LIST_TOOLBAR`、`LIST_ROW_BUTTON`、`HEADER_TOOLBAR`、`DETAIL_ROW_BUTTON`。
- 新增按钮时，系统会自动给 admin 岗位补齐可用权限。

### 4.2.6 `sys_module_state`

作用：定义状态型模块的状态集合。

关键字段：

| 字段 | 说明 |
| --- | --- |
| `module_id` | 模块 ID |
| `state_code` | 状态编码 |
| `state_name` | 状态名称 |
| `is_initial` | 是否初始状态 |
| `is_final` | 是否最终状态 |
| `sort` | 排序 |

关键索引：

- `uk_sys_module_state`
- `idx_sys_module_state_sort`

说明：

- 当前种子数据中采购订单、销售订单都内置了 `unaudited`、`pre_audited`、`audited`、`shipped`、`completed` 五个状态。
- 该表当前处于“可查询、可用于状态权限配置”的状态，但没有单独的 CRUD 接口。

### 4.2.7 `sys_module_transition`

作用：定义状态流转。

关键字段：

| 字段 | 说明 |
| --- | --- |
| `module_id` | 模块 ID |
| `action_button_code` | 触发按钮编码 |
| `from_state_code` | 来源状态编码 |
| `to_state_code` | 目标状态编码 |
| `sort` | 排序 |

关键索引：

- `uk_sys_module_transition`
- `idx_sys_module_transition_state`

说明：

- 当前主要作为模块详情和状态权限页的结构数据来源。
- 还未开放维护接口。

## 6.6 接口鉴权方式

当前接口统一通过 `@PreAuthorize` + `@moduleAccess` 完成鉴权：

- `@moduleAccess.hasModule(moduleCode)`
  - 判断用户是否有模块可见权限。
- `@moduleAccess.hasButton(moduleCode, buttonCode)`
  - 判断用户是否有按钮可用权限，要求权限级别 `>= 2`。

这意味着：

- 页面能否进入，取决于模块可见性。
- 操作按钮能否执行，取决于按钮权限。
- 系统管理不再依赖传统 `sys_menu` / `sys_role_menu` 下发接口权限。

## 7. 各模块详细设计

## 7.1 模块管理

### 7.1.1 模块作用

模块管理是整个平台的主数据中心，负责维护：

- 模块树
- 模块基本信息
- Tab 定义
- 字段定义
- 按钮定义
- 状态、流转的查询载体
- 模块结构预览

### 7.1.2 数据依赖

- 主表：`sys_module`
- 子表：`sys_module_tab`
- 子表：`sys_module_field`
- 子表：`sys_module_field_title`
- 子表：`sys_module_button`
- 子表：`sys_module_state`
- 子表：`sys_module_transition`

### 7.1.3 关键实现规则

1. 模块树按 `parent_id + sort` 组装。
2. 查询模块详情时，按模块聚合：
   - `headerTabs`
   - `detailTabs`
   - `buttons`
   - `states`
   - `transitions`
3. 新增字段时，自动初始化 `sys_module_field_title` 默认记录。
4. 新增模块、字段、按钮时，自动给 admin 所在岗位补齐权限。
5. 任何结构变化都会递增 `context_version`。

### 7.1.4 前端页面行为

当前 `module-center` 页面是“树摘要 + 详情懒加载”模型：

- 左侧只加载模块树。
- 选中模块后再查询详情。
- 工作区展示模块信息、Tab、字段、按钮。
- 状态和流转目前不在该页面独立编辑。

### 7.1.5 现状边界

- `context-preview` 只用于预览结构，不注入岗位和状态上下文。
- 状态与流转是结构数据的一部分，但当前没有独立维护能力。

## 7.2 岗位管理

### 7.2.1 模块作用

岗位管理的本质不是只维护组织岗位，而是为整个权限平台提供“授权主体”。

它负责：

- 组织单元维护
- 岗位维护
- 组织岗位树组装
- 岗位详情查询
- 岗位员工绑定与解绑
- 给功能授权提供岗位树和岗位详情

### 7.2.2 数据依赖

- `sys_org_unit`
- `sys_org_post`
- `sys_user_org_post`
- `sys_user`

### 7.2.3 关键实现规则

1. 树结构同时包含组织节点和岗位节点。
2. 新增岗位时，如果有父岗位，则岗位归属组织自动继承父岗位组织。
3. 岗位不支持跨组织迁移。
4. 删除岗位前必须没有子岗位、没有岗位员工。
5. 岗位员工查询和候选员工查询全部按分页处理。

### 7.2.4 权限平台中的作用

岗位管理在权限平台中的定位有两层：

- 业务配置层：给功能授权提供岗位树、岗位详情。
- 运行鉴权层：通过 `sys_user_org_post` 决定当前用户可继承哪些岗位权限。

## 7.3 功能授权

### 7.3.1 模块作用

功能授权负责维护岗位对模块的“基线权限”。

这层权限决定：

- 这个模块岗位能否看见
- 这个模块能否出现在导航
- 模块下每个字段是隐藏、只读还是可写
- 模块下每个按钮是隐藏、禁用还是可用

### 7.3.2 数据依赖

- 结构来源：`sys_module`、`sys_module_tab`、`sys_module_field`、`sys_module_button`
- 授权表：`sys_org_post_module_auth`
- 授权表：`sys_org_post_field_auth`
- 授权表：`sys_org_post_button_auth`

### 7.3.3 关键实现规则

1. 页面详情通过 `orgPostId + moduleId` 聚合得到。
2. 保存时先校验岗位和模块存在。
3. 保存时先删除当前岗位在该模块下的旧权限，再全量重建。
4. 若请求中未带某字段或某按钮的权限值，则默认写为 `0`。
5. 若模块不可见，则导航可见自动收敛为不可见。
6. 保存后模块 `context_version` 递增。

### 7.3.4 数据返回形态

功能授权页面返回：

- `visibleModuleCount`
- `postDetail`
- `moduleDetail`
- `config`

这种设计让前端可以在一个接口中同时拿到：

- 当前岗位信息
- 当前模块信息
- 当前岗位对该模块的基线授权

### 7.3.5 前端作用

前端 `function-auth` 页面：

- 左侧岗位树来自岗位管理能力。
- 中间模块树来自模块管理能力。
- 右侧按岗位 + 模块配置基线权限。

这说明功能授权是“岗位管理”和“模块管理”的汇合页。

## 7.4 字段标题

### 7.4.1 模块作用

字段标题用于处理“字段展示文案”和“字段技术定义”分离的问题。

它解决的核心问题是：

- 技术字段编码要稳定。
- 展示标题可能随业务语言调整。
- 占位提示和帮助文案属于展示层，不应硬编码在字段定义中。

### 7.4.2 数据依赖

- 结构来源：`sys_module`、`sys_module_tab`、`sys_module_field`
- 覆盖表：`sys_module_field_title`

### 7.4.3 关键实现规则

1. 页面详情继续复用模块详情。
2. 字段标题实际保存在 `sys_module_field_title`。
3. 保存时按 `moduleId` 批量处理字段标题。
4. 当前语言固定使用 `zh_CN`。
5. 若用户输入标题与 `default_title` 一致，则覆盖值落空，回退默认标题。
6. 保存后模块 `context_version` 递增。

### 7.4.4 设计收益

- 模块结构稳定，展示文案可灵活调整。
- 后续支持多语言时，可以继续沿用 `field_id + locale` 结构扩展。

## 7.5 状态权限

### 7.5.1 模块作用

状态权限负责给“状态型模块”增加状态维度的附加约束。

它的定位不是重新做一套权限，而是对岗位基线权限做进一步收紧。

### 7.5.2 数据依赖

- 结构来源：`sys_module`、`sys_module_state`、`sys_module_transition`、`sys_module_field`、`sys_module_button`
- 限制表：`sys_module_state_field_auth`
- 限制表：`sys_module_state_button_auth`

### 7.5.3 关键实现规则

1. 只允许对 `stateful_flag = 1` 的模块配置状态权限。
2. 保存时先删除该模块的全部状态权限，再按请求重建。
3. 若某字段或按钮的状态限制值为 `2`，表示“不收紧”，不会落库。
4. 若数据库没有状态限制记录，则运行时默认视为 `2`。
5. 保存后模块 `context_version` 递增。

### 7.5.4 页面形态

前端 `state-auth` 页面：

- 左侧只显示状态型模块树。
- 右侧按状态分组配置字段和按钮限制。

### 7.5.5 设计收益

- 岗位基线权限负责岗位职责边界。
- 状态权限负责业务流转阶段边界。
- 两者分离后，配置职责更清晰。

## 8. 导航与控制器鉴权设计

## 8.1 导航生成

当前前端动态路由不再来源于 `sys_menu`，而是来源于：

- 后端接口：`GET /sys/module/nav-tree`
- 前端 Store：`permission.ts`

处理过程：

1. 后端根据用户岗位，查询可见且可导航的模块。
2. 前端把模块树转换为 `vue-router` 路由。
3. 目录型模块挂载 `Layout`。
4. 页面型模块根据 `component_path` 动态加载实际页面。

这意味着“导航是否显示”由模块权限平台直接控制。

## 8.2 控制器鉴权

当前后端控制器使用 `SysModuleAccessEvaluator` 做表达式鉴权：

- `hasModule(moduleCode)`：判断模块可见
- `hasButton(moduleCode, buttonCode)`：判断按钮权限级别是否达到可操作

其底层查询逻辑来自：

- `selectUserVisibleNavModuleIdList`
- `countUserVisibleModule`
- `selectUserButtonPermissionLevel`

这样做的结果是：

- 页面入口权限
- 控制器操作权限
- 后台导航权限

全部统一收敛到了模块权限平台。

## 11. 设计总结

当前模块权限平台的整体设计可以概括为：

- 岗位管理负责“谁被授权”。
- 模块管理负责“授权对象是什么”。
- 功能授权负责“岗位基线能做什么”。
- 字段标题负责“字段展示成什么样”。
- 状态权限负责“在不同状态下再收紧到什么程度”。

这种设计的优点是：

- 授权主体清晰。
- 主数据与展示数据解耦。
- 岗位权限与状态权限解耦。
- 导航、按钮和页面结构统一由模块平台驱动。
- 便于后续扩展运行时上下文装配、多语言标题和更多业务模块。

当前需要注意的边界是：

- 状态与流转仍偏元数据查询态，维护能力未完全开放。
- 运行时完整权限上下文装配尚未单独抽象为业务服务。
- 权限保存采用“先删后建”的整模块覆盖模式，适合当前后台管理场景，但并发编辑时需要额外注意覆盖风险。
