# Soli Admin 技术架构说明

## 1. 架构总览

Soli Admin 采用前后端分离架构：

- 前端：Vue 3 + Vite + TypeScript + Pinia + Element Plus
- 后端：Java 21 + Spring Boot 3 + Spring Security + MyBatis + PageHelper + MapStruct
- 数据库：MySQL 兼容数据库

整体设计不是“页面驱动后端”，而是“模块模型驱动页面和权限”。后端围绕模块上下文构建统一描述，前端按照上下文渲染字段、按钮和状态行为。

## 2. 后端模块拆分

后端聚合工程位于 [soli-admin-backend/pom.xml](../soli-admin-backend/pom.xml)，当前拆分为四层：

### 2.1 `soli-admin-common`

公共能力层，负责通用基础设施：

- `soli-admin-common-api`
  - 通用返回对象、分页对象、异常对象
- `soli-admin-common-core`
  - 通用实体基类
  - Swagger 配置
  - ID 生成配置
  - 公司上下文持有器 `CompanyContextHolder`
- `soli-admin-common-web`
  - 安全过滤器
  - JWT 访问过滤
  - 全局异常处理
  - 通用 `BaseController`

### 2.2 `soli-admin-framework`

平台核心能力层：

- `soli-admin-auth-api/core/web`
  - 登录认证、验证码、Token 生成
- `soli-admin-system-api/core/web`
  - 用户、岗位、字典、参数、模块、字段标题、状态权限、功能授权等系统能力

### 2.3 `soli-admin-business`

业务模块层：

- `soli-admin-business-api`
  - 采购订单对外对象和服务接口
- `soli-admin-business-core`
  - 采购订单 Mapper、Entity、ServiceImpl、权限校验器
- `soli-admin-business-web`
  - 采购订单控制器

### 2.4 `soli-admin-start`

启动聚合层，位于 [soli-admin-start/pom.xml](../soli-admin-backend/soli-admin-start/pom.xml)，负责把认证模块、系统模块、业务模块统一装配成一个可运行应用。

## 3. 前端模块拆分

前端工程位于 [soli-admin-frontend/package.json](../soli-admin-frontend/package.json)，技术栈如下：

- Vue 3.5
- Vite 7
- TypeScript 5.9
- Pinia 3
- Vue Router 5
- Element Plus 2

当前前端已经接入后端模块上下文，典型特点是：

- 页面通过 `/context` 接口获取模块上下文
- 字段和按钮统一从 `fieldConfigs` 读取
- 不再由前端自行拼装字段权限和状态权限
- 采购订单已经从原来的模板式模块迁移为独立业务模块

## 4. 核心运行链路

### 4.1 登录链路

1. 前端调用 `/auth/login-using-username`
2. 后端认证成功后返回 `TokenDTO`
3. 前端后续请求携带 `Authorization`
4. `AccessTokenFilter` 解析用户 ID、公司 ID
5. 公司 ID 放入 `CompanyContextHolder`
6. 控制器和权限表达式从上下文读取当前公司

### 4.2 页面渲染链路

1. 前端进入某个业务页面
2. 页面请求对应模块的 `/context`
3. 后端通过 `SysModuleContextService` 构建 `SysModuleContextDTO`
4. 上下文里返回：
   - 模块基础信息
   - 当前状态信息
   - `fieldConfigs`
5. 前端按照 `fieldConfigs` 渲染字段和按钮

### 4.3 权限链路

权限分两层叠加：

- 岗位权限
  - 决定模块、字段、按钮的基础权限
- 状态权限
  - 决定在某个状态下字段和按钮还能否继续可见、可编辑、可用

最终权限计算规则可理解为：

`最终权限 = 岗位权限 ∩ 状态权限`

在代码里，核心实现位于 [SysModuleContextServiceImpl.java](../soli-admin-backend/soli-admin-framework/soli-admin-system-core/src/main/java/com/soli/system/core/service/impl/sysmodule/SysModuleContextServiceImpl.java)。

## 5. 表结构分析

表结构定义位于 [init_Table.sql](../soli-admin-backend/sql/init_Table.sql)。当前可以分为五个域。

### 5.1 用户与组织域

#### `sys_user`

用户主表，保存：

- 登录账号
- 密码
- 昵称
- 手机、邮箱、头像
- 用户类型
- 状态

#### `sys_org_unit`

组织单元表，采用树形结构，关键字段：

- `parent_id`
- `ancestors`
- `org_code`
- `org_name`
- `org_type`

#### `sys_org_post`

岗位表，同样支持树结构，关键字段：

- `org_unit_id`
- `parent_post_id`
- `ancestors`
- `post_code`
- `post_name`
- `manager_user_id`

#### `sys_user_org_post`

用户与岗位关联表，负责把用户挂到岗位上，并区分主岗位和兼岗。

### 5.2 模块建模域

#### `sys_module`

模块主表，是整个平台的核心入口，定义：

- 模块编码、名称、类型
- 路由路径
- 前端组件路径
- 是否导航可见
- 是否状态型模块
- 状态字段编码
- 上下文版本

#### `sys_module_component`

模块组件定义表。一个模块可拆成多个组件，例如：

- `user_form`
- `org_form`
- `post_form`
- `detail`

#### `sys_module_field`

模块字段定义表，负责描述字段基础定义，例如：

- 字段编码
- 默认标题
- 组件类型
- 数据路径
- 值类型
- 是否必填
- 排序

这个表现在更偏“字段定义源”，而不是运行时直接使用表。

#### `sys_module_field_title`

字段标题表，是当前运行时非常关键的表。它把字段运行时需要的信息完整保留下来，包括：

- `field_id`
- `module_id`
- `component_id`
- `component_code`
- `field_code`
- `default_title`
- `display_title`
- `placeholder`
- `help_text`
- `component_type`
- `data_path`
- `value_type`
- `required_flag`

这张表的意义是：

- 支持字段标题独立维护
- 支持运行时直接读取字段标题配置
- 避免再依赖模块字段表做联表补齐

#### `sys_module_button`

模块按钮定义表，描述按钮编码、标题、排序、状态。

#### `sys_module_state`

模块状态定义表，适用于状态型模块，描述：

- 状态编码
- 状态名称
- 是否初始状态
- 是否最终状态

#### `sys_module_transition`

状态流转表，描述从哪个状态，通过哪个按钮，流转到哪个状态。

### 5.3 权限域

#### `sys_org_post_module_auth`

岗位与模块权限表，决定岗位是否能看到模块和导航。

#### `sys_org_post_field_auth`

岗位字段权限表，`permission_level` 用于控制字段：

- 0 隐藏
- 1 只读
- 2 可写

#### `sys_org_post_button_auth`

岗位按钮权限表，`permission_level` 用于控制按钮：

- 0 隐藏
- 1 禁用
- 2 可用

### 5.4 状态权限域

#### `sys_module_state_field_auth`

状态字段权限表，用来限制某个状态下字段还能不能编辑。

#### `sys_module_state_button_auth`

状态按钮权限表，用来限制某个状态下按钮还能不能点击。

### 5.5 业务单据域

#### `purchase_order`

采购订单主表，保存单据头信息：

- 公司
- 单号
- 单据日期
- 供应商
- 结算方式
- 仓库
- 币种
- 状态
- 创建人、审核人
- 金额汇总

#### `purchase_order_item`

采购订单明细表，保存物料、数量、单价、税率、金额等。

#### `purchase_order_source`

采购订单来源单据表，用来挂接来源业务单据。

#### `purchase_order_attachment`

采购订单附件表。

#### `purchase_order_activity`

采购订单操作日志表，记录动作编码、动作名称、操作人、操作时间和日志内容。

### 5.6 关键关系总结

- 一个用户可关联多个岗位
- 一个岗位可授权多个模块
- 一个模块包含多个组件
- 一个组件包含多个字段
- 一个模块包含多个按钮和多个状态
- 字段权限和按钮权限按岗位、状态两套体系共同约束
- 一个采购订单对应多条明细、多条来源、多条附件、多条日志

## 6. Java 对象分析

Java 对象整体分成五类。

### 6.1 请求与查询对象

位于 `api` 模块，负责承接前端参数。

典型对象：

- `SysUserCreateRequest`
- `SysUserModifyRequest`
- `SysUserQuery`
- `PurchaseOrderSaveRequest`
- `PurchaseOrderPageQuery`

特点：

- 与接口输入强绑定
- 带校验注解
- 不直接承担数据库映射责任

### 6.2 数据传输对象

数据传输对象用于服务层和控制器之间传递数据。

典型对象：

- `SysModuleContextDTO`
- `SysUserDTO`
- `SysOrgPostDetailDTO`
- `PurchaseOrderDetailDTO`
- `PurchaseOrderPageDTO`

其中 [SysModuleContextDTO.java](../soli-admin-backend/soli-admin-framework/soli-admin-system-api/src/main/java/com/soli/system/service/sysmodule/SysModuleContextDTO.java) 是当前平台的关键对象，核心字段包括：

- `moduleId`
- `moduleCode`
- `state`
- `fieldConfigs`

`fieldConfigs` 已经扁平化，采用 `组件:code` 作为键，例如：

- `user_form:username`
- `detail:qty`
- `button:save`

这样前端就不需要再处理 `headerTabs`、`detailsTabs` 等旧结构。

### 6.3 实体对象

实体对象位于 `core` 模块，直接映射数据库结构。

典型对象：

- `SysUserEntity`
- `SysOrgPostEntity`
- `SysModuleEntity`
- `PurchaseOrderEntity`
- `PurchaseOrderItemEntity`

例如 [PurchaseOrderEntity.java](../soli-admin-backend/soli-admin-business/soli-admin-business-core/src/main/java/com/soli/business/core/service/impl/purchaseorder/PurchaseOrderEntity.java) 就是采购订单主表的 Java 映射对象。

### 6.4 查询模型对象

查询模型对象主要用于承接 Mapper 查询结果，一般用于：

- 树节点
- 联表详情
- 统计结果
- 权限结果

典型对象：

- `SysModuleTreeNodeModel`
- `SysOrgPostDetailModel`
- `SysUserModuleFieldPermissionModel`
- `SysUserModuleButtonPermissionModel`
- `PurchaseOrderOverviewModel`

这类对象的特点是更贴近 SQL 查询结果，而不一定直接暴露给前端。

### 6.5 转换器对象

当前系统使用转换器负责请求对象、数据传输对象和实体对象之间的转换。

典型类：

- `SysUserConverter`
- `SysConfigConverter`
- `SysDictConverter`
- `SysModuleConverter`
- `SysOrgPostConverter`

这样能避免控制器直接操作 Entity，也便于后续扩展字段。

## 7. 服务层分析

### 7.1 通用增删改查服务

系统模块大量场景采用：

- `BaseCrudMapper`
- `BaseCrudServiceImpl`

这一套用于字典、配置、用户等标准管理模块，能统一：

- 新增
- 修改
- 删除
- 分页
- 钩子扩展

### 7.2 模块上下文服务

模块上下文服务接口位于 [SysModuleContextService.java](../soli-admin-backend/soli-admin-framework/soli-admin-system-api/src/main/java/com/soli/system/service/sysmodule/SysModuleContextService.java)。

实现位于 [SysModuleContextServiceImpl.java](../soli-admin-backend/soli-admin-framework/soli-admin-system-core/src/main/java/com/soli/system/core/service/impl/sysmodule/SysModuleContextServiceImpl.java)。

它负责：

- 根据模块编码定位模块
- 拉取模块详情
- 计算当前状态
- 读取岗位字段权限和按钮权限
- 叠加状态字段权限和按钮权限
- 输出统一的 `fieldConfigs`

这是当前系统最核心的平台服务之一。

### 7.3 采购订单服务

采购订单服务实现位于 [PurchaseOrderServiceImpl.java](../soli-admin-backend/soli-admin-business/soli-admin-business-core/src/main/java/com/soli/business/core/service/impl/purchaseorder/PurchaseOrderServiceImpl.java)。

它承担的职责包括：

- 分页查询
- 详情查询
- 新增单据
- 修改单据
- 执行动作
- 计算金额
- 写入操作日志

它是一个比较典型的业务服务实现样板。

## 8. Mapper 层分析

Mapper 层主要分为三类：

### 8.1 通用增删改查 Mapper

- `BaseCrudMapper`
- `SysUserMapper`
- `SysDictMapper`
- `SysDictDataMapper`
- `SysConfigMapper`

适合标准管理模块。

### 8.2 平台建模与权限 Mapper

- `SysModuleMapper`
- `SysModuleTitleMapper`
- `SysModulePermissionMapper`
- `SysOrgPostMapper`

这些 Mapper 负责平台最核心的数据装配：

- 模块树
- 组件与字段定义
- 字段标题
- 岗位权限
- 状态权限
- 组织与岗位树

### 8.3 业务 Mapper

- `PurchaseOrderMapper`

负责采购订单主表、明细、来源、附件、日志的持久化访问。

这使得采购订单的权限控制不仅看用户角色，还会看单据当前状态。

## 10. 权限表达式分析

当前项目采用 Spring Security `@PreAuthorize` 表达式做权限拦截。

### 10.1 模块级

- `@moduleAccess.hasModule('sys_user')`

用于控制页面或模块是否可见。

### 10.2 按钮级

- `@moduleAccess.hasButton('sys_org_post', 'userBind')`

用于控制非状态型按钮。

### 10.3 状态型按钮

- `@moduleAccess.hasStateButton('sys_user', 'modify', 'edit')`
- `@purchaseOrderAccess.hasOrderButton('save', #request.id)`

用于根据状态控制可编辑性和按钮动作。

## 11. 模块上下文结构分析

当前模块上下文已经统一收敛为一份结构，不再区分 `headerTabs`、`detailsTabs`。

核心结构：

- `moduleId`
- `moduleCode`
- `state`
- `fieldConfigs`

`fieldConfigs` 同时放字段和按钮：

- 字段示例：`user_form:username`
- 按钮示例：`button:create`

每个配置项统一包含：

- 标识信息
- 标题信息
- 组件信息
- 数据路径
- 权限级别
- 可见、可编辑、是否禁用

这种结构的优势是：

- 前端消费成本低
- 后端输出一致
- 便于公共组件抽象

## 12. 二次开发建议

如果要继续扩展业务模块，建议按下面步骤：

1. 在 `sys_module` 及其子表补齐模块、组件、字段、按钮、状态定义
2. 在字段标题表生成运行时字段标题数据
3. 在岗位权限和状态权限表配置默认规则
4. 在 `business-api` 定义 Request、DTO、Service 接口
5. 在 `business-core` 定义 Entity、Mapper、ServiceImpl、权限校验器
6. 在 `business-web` 增加 Controller
7. 前端页面统一对接 `/context`
8. 页面字段和按钮全部改为读取 `fieldConfigs`

## 13. 结论

Soli Admin 的技术架构不是简单的三层 CRUD，而是“平台建模 + 权限引擎 + 业务模块”的组合架构。

它最大的技术特点有三点：

- 模块建模完整
- 权限粒度足够细
- 前后端围绕模块上下文协作

这套架构非常适合企业中后台长期演进，也适合在采购、单据、审批、主数据等复杂业务场景中持续扩展。
