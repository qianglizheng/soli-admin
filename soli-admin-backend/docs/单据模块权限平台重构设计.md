# 单据模块权限平台重构设计

## 1. 文档说明

本文档以当前仓库内已经落地的实现为准，覆盖以下范围：

- 模块管理
- 功能授权
- 字段标题
- 状态权限
- 初始化建表与种子数据

不再沿用旧版文档中的 `/admin/*` 接口示例，也不再描述前端 mock 方案。当前前端页面已经全部切换为后端数据驱动。

## 2. 当前实现结论

### 2.1 平台拆分

当前后台管理端围绕“模块元数据 + 岗位基线权限 + 状态收紧权限”拆分为 4 个核心页面：

1. 模块管理
2. 功能授权
3. 字段标题
4. 状态权限

岗位主数据仍由现有岗位管理能力承载，权限页只消费岗位树和岗位详情，不负责编辑岗位主数据。

### 2.2 权限模型

当前实现的权限边界如下：

- 模块管理维护模块、Tab、字段、按钮、状态、流转等元数据
- 字段标题维护字段展示元数据：`displayTitle`、`placeholder`、`helpText`
- 功能授权维护岗位基线权限
- 状态权限只维护状态维度的附加限制，只允许“收紧”，不允许“放大”
- 系统后台导航与控制器鉴权也已经切到模块平台，不再依赖 `sys_menu`

统一权限级别如下：

- 字段权限：`0 = 隐藏`，`1 = 只读`，`2 = 可写`
- 按钮权限：`0 = 隐藏`，`1 = 禁用`，`2 = 可用`
- 状态字段限制：`0 = 隐藏`，`1 = 只读`，`2 = 不收紧`
- 状态按钮限制：`0 = 隐藏`，`1 = 禁用`，`2 = 不收紧`

最终运行时权限计算规则仍为：

- `finalFieldLevel = min(orgPostFieldLevel, stateFieldLimitLevel)`
- `finalButtonLevel = min(orgPostButtonLevel, stateButtonLimitLevel)`

当状态权限表中不存在记录时，默认按 `2 = 不收紧` 处理，即直接继承岗位基线结果。

### 2.3 状态模型

当前种子数据内置了 5 个标准状态：

- `unaudited`
- `pre_audited`
- `audited`
- `shipped`
- `completed`

`create` 不再作为独立状态处理，初始单据统一归入 `unaudited`。

## 3. 页面实现说明

### 3.1 模块管理

模块管理当前职责：

- 维护模块树
- 维护模块基础信息
- 维护 Tab、字段、按钮
- 维护状态与流转定义
- 提供上下文预览

当前模块详情接口返回的数据已经包含：

- `headerTabs`
- `detailTabs`
- `buttons`
- `states`
- `transitions`

字段详情接口会回填以下可展示元数据：

- `defaultTitle`
- `displayTitle`
- `placeholder`
- `helpText`

说明：

- `GET /sys/module/{id}/context-preview` 当前仅用于模块管理做结构预览
- 该接口不再注入 mock 的当前岗位或当前状态
- 当前返回的是模块结构上下文，不是完整运行时权限上下文
- Tab 仍以 `sys_module_tab` 为唯一主数据来源，不再为字段标题或功能授权额外复制一套 Tab 表，避免结构冗余
- 新增字段时自动写入 `sys_module_field_title` 默认记录；新增模块、字段、按钮时自动补齐 admin 岗位权限
- `GET /sys/module/nav-tree` 返回当前登录用户可见的全部导航模块树，前端侧边栏主菜单已完全切到该接口
- 采购列表页等显式菜单路由已经由模块数据驱动，创建页/详情页等隐藏路由仍由前端本地作为扩展子路由挂载

### 3.2 功能授权

功能授权页当前全部使用后端数据：

- 左侧岗位树来自后端岗位树接口
- 中间模块树来自后端模块树接口
- 右侧配置区通过 `orgPostId + moduleId` 查询岗位基线权限
- 保存时回写岗位模块权限、字段权限、按钮权限

前端不再依赖本地 mock 生成岗位权限数据。

### 3.3 字段标题

字段标题当前全部使用后端数据：

- 模块树来自后端
- 模块详情继续复用模块管理详情数据，但字段展示文案来自独立表 `sys_module_field_title`
- 保存时批量更新 `sys_module_field_title.display_title / placeholder / help_text`
- 字段标题表只保存覆盖值；`display_title` 为空时回退到 `sys_module_field.default_title`

前端不再维护本地标题 mock 仓库。

### 3.4 状态权限

状态权限当前全部使用后端数据：

- 左侧只展示 `stateful_flag = 1` 的模块树
- 页面详情直接返回模块详情 + 当前模块状态限制配置
- 保存时批量覆盖当前模块的状态字段限制、状态按钮限制

实现约束：

- 只保存“收紧”结果
- `permission_level >= 2` 的状态限制不会落库
- 因此数据库里只保存状态下真正被收紧的资源

## 4. 数据库设计

### 4.1 已落地表

当前 `sql/init_Table.sql` 已包含以下与本次重构直接相关的表：

### 4.2 关键字段

#### `sys_module_field`

当前仅保留字段结构与默认标题：

- `default_title`
- `component_type`
- `data_path`

#### `sys_module_field_title`

- `field_id`
- `locale`

- `placeholder`

#### `sys_module_state`

- `state_code`

#### `sys_module_transition`

- `sys_org_post_module_auth` 保存模块是否可见、导航是否可见

## 5. 初始化数据

### 5.1 已包含建表语句

是，当前仓库已经包含完整建表语句，位置如下：

- `soli-admin-backend/sql/init_Table.sql`

本次重构新增和使用的表已经写入该文件，不需要再单独补表。

### 5.2 已包含初始化数据

当前仓库已经包含对应种子数据，位置如下：

- `soli-admin-backend/sql/init_Data.sql`

种子数据覆盖了以下内容：

- 组织与岗位树
- 模块、Tab、字段、按钮
- 标准状态与流转
- admin 用户主岗位映射
- admin 岗位的模块、字段、按钮全量基线权限
- 系统管理导航模块与按钮定义由启动引导自动补齐

### 5.3 admin 权限处理

当前实现已经把模块权限平台相关权限放给 admin 用户，具体包括两层：

#### 模块导航 / 控制器权限

当前初始化逻辑中：

- `SysManagementModuleBootstrap` 会在启动时补齐系统管理导航模块树
- 控制器 `@PreAuthorize` 已改为按模块可见性、按钮可用性判定
- 不再通过 `sys_menu` / `sys_role_menu` 下发接口权限码

#### 岗位基线权限

当前初始化逻辑中：

- admin 用户通过 `sys_user_org_post` 绑定到岗位 `9101`
- 启动时会为 admin 关联岗位补齐当前库内所有模块的可见权限
- 启动时会为 admin 关联岗位重建当前库内所有字段 `2 = 可写`
- 启动时会为 admin 关联岗位重建当前库内所有按钮 `2 = 可用`
- 模块管理后续新增模块、字段、按钮时，也会自动为 admin 绑定岗位补齐默认权限
- 模块管理后续新增字段时，也会自动补齐 `sys_module_field_title` 默认记录

说明：

- 当前没有为状态权限表预置收紧数据
- 这样 admin 默认不会被状态权限额外限制

## 7. 前端对接方式

当前前端已移除模块权限平台相关的 mock 文件，页面全部改为调用真实接口。

对应 API 文件如下：

- `soli-admin-frontend/src/api/moduleCenter.ts`
- `soli-admin-frontend/src/api/functionAuth.ts`
- `soli-admin-frontend/src/api/moduleTitle.ts`
- `soli-admin-frontend/src/api/stateAuth.ts`

页面数据装配方式如下：

- 功能授权页把后端返回的 `fieldId/buttonId` 权限列表映射为前端按 `fieldCode/buttonCode` 管理的表单结构
- 状态权限页把后端返回的 `permissionsByState` 映射为按 `stateCode -> fieldCode/buttonCode` 管理的编辑结构
- 字段标题页直接基于模块详情中的字段定义编辑 `displayTitle/placeholder/helpText`
- 功能授权和状态权限只保存资源 ID 与权限级别，不复制模块结构字段，页面展示时再按 `fieldId/buttonId` 回关联资源主数据

说明：

- 前端页面不再从本地 fixture 组装模块、岗位、状态权限
- 页面中的预览抽屉也是基于后端数据即时组装，不再读取 mock 上下文

## 8. 当前实现边界

以下内容当前未做成完整运行时能力，文档中需要明确区分：

### 8.1 已实现

- 模块管理元数据维护
- 岗位基线权限维护
- 字段标题覆盖维护
- 状态收紧权限维护

### 8.2 未实现为统一运行时接口

当前还没有落地“按用户当前岗位 + 单据当前状态 + 字段标题覆盖”统一合并后的运行时接口。

现状是：

- 模块管理有 `GET /sys/module/{id}/context-preview`
- 该接口只做模块结构预览
- 不负责拼装真实岗位权限
- 不负责拼装真实状态限制
- 不负责根据业务单据主键推导当前状态

如果后续要接入真实运行时渲染，需要在此基础上新增独立的运行时上下文服务。

## 9. 本次重构后的约束

后续继续开发时，按以下原则执行：

1. 模块结构只在模块管理维护。
2. 字段展示文案只在字段标题维护。
3. 岗位基线权限只在功能授权维护。
4. 状态附加限制只在状态权限维护。
5. 状态权限只能收紧，不能放大岗位基线。
6. 前端页面不得再回退到本地 mock 数据。
7. Tab 不做额外镜像表，字段标题与权限页统一通过模块主数据关联 Tab 结构。
8. 模块新增字段、按钮、模块时，需要同步维护字段标题默认记录与 admin 权限默认记录。
