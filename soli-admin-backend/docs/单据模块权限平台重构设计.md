# 单据模块权限平台重构设计

## 1. 本版结论

本版设计以当前前端原型为准，不兼容旧的“角色 + 菜单 + perms”模式去承载单据内部权限，也不再使用 `bill_template` 相关表命名。

核心结论如下：

- 权限平台统一升级为“模块化权限平台”，面向所有单据模块复用，不为某一个单据模板单独建模。
- 后台管理端拆分为 5 个工作台：`岗位管理`、`模块中心`、`功能授权`、`字段标题中心`、`状态权限中心`。
- `岗位管理` 负责组织岗位主数据和岗位员工关系维护，不再把岗位编辑动作塞进 `功能授权`。
- `功能授权` 只负责岗位基线权限，不出现状态选择器，不配置状态权限。
- `状态权限中心` 只负责状态附加限制，且只能在岗位基线之上收紧，不能放大权限。
- 运行时权限计算顺序固定为：先岗位，后状态。
- `create` 不再作为独立状态，单据初始状态统一归入 `未审核`。
- 标准状态固定为 5 个：`unaudited`、`pre_audited`、`audited`、`shipped`、`completed`。
- 前端进入模块时先获取模块上下文，字段标题、字段权限、按钮权限都从上下文中读取，前端不再散落写权限判断。

## 2. 与当前前端原型对齐

### 2.1 岗位管理

当前前端已按独立页面原型落地：

- 右侧为岗位树。
- 岗位树节点同时承载集团、总公司、分公司、岗位层级。
- 右侧支持：`新增岗位`、`新增下级`、`编辑岗位`。
- 左侧为当前岗位工作区。
- 左侧支持：`添加员工`、`查看员工`，并展示岗位员工预览。

这意味着岗位主数据维护已经从功能授权页面剥离，形成独立工作台。

### 2.2 模块中心

模块中心继续负责模块元数据维护：

- 左侧为模块树。
- 右侧固定为模块概览 + 四个工作页签。
- 四个页签分别是：
  - 基础信息
  - 单头设计
  - 明细设计
  - 按钮设计

模块中心维护的内容是运行时上下文的结构来源，不承担岗位或状态授权逻辑。

### 2.3 功能授权

功能授权页面按本版方案收敛为“三段式”结构：

- 左侧：岗位树，仅用于选择授权主体。
- 右侧上半部分：当前岗位摘要、保存、恢复、岗位基线预览。
- 右侧主体：模块树 + 权限编辑区。

权限编辑区保留三个页签：

1. 模块授权
2. 按钮权限
3. 字段权限

本页只配置岗位基线，不配置状态权限，不出现状态选择器。

### 2.4 状态权限中心

状态权限中心独立存在：

- 左侧仅展示状态型模块。
- 右侧按状态维护字段与按钮限制。
- 页面只做“收紧”配置，不做基线放大。

## 3. 核心设计原则

### 3.1 授权主体是组织岗位，不是角色

角色继续保留在平台入口层，负责：

- 登录态
- 菜单入口
- 通用接口访问

单据模块内部权限改由“组织岗位”负责。岗位是带组织归属的岗位实例，例如：

- 总公司 / 采购员
- 总公司 / 预审岗
- 总公司 / 审核岗
- 华东分公司 / 销售经理
- 华南分公司 / 运营中心

同名岗位在不同公司下可以拥有不同权限。

### 3.2 岗位主数据管理与岗位授权分离

为避免页面职责混乱，岗位相关能力拆成两部分：

- `岗位管理`
  - 维护岗位树
  - 维护岗位基本信息
  - 维护岗位挂靠员工
- `功能授权`
  - 消费岗位树
  - 为岗位配置模块基线权限

因此：

- 岗位的新增、编辑、员工分配不在功能授权页面处理。
- 功能授权页面只处理“岗位 -> 模块 -> 字段/按钮”这一层授权关系。

### 3.3 先岗位，后状态

统一计算规则：

1. 先读取岗位基线权限。
2. 再读取当前单据状态的附加限制。
3. 状态权限只允许收紧，不允许放大。

统一权限级别：

- 字段：`0 = hidden`、`1 = read`、`2 = write`
- 按钮：`0 = hidden`、`1 = disabled`、`2 = enabled`

最终计算：

- `finalFieldLevel = min(orgPostFieldLevel, stateFieldLevel)`
- `finalButtonLevel = min(orgPostButtonLevel, stateButtonLevel)`

如果状态未配置，则直接继承岗位基线。

### 3.4 模块上下文优先

前端进入模块后，先获取模块上下文，再决定渲染。

模块上下文必须同时返回：

- 模块基础信息
- 单头 Tab 分组
- 明细 Tab 分组
- 字段标题
- 字段权限
- 按钮权限
- 当前岗位
- 当前状态

前端不再允许散落写：

- `if (roleCode === ...)`
- `if (postCode === ...)`
- `if (state === ...)`
- `if (fieldCode === ...)`

## 4. 后台管理模块划分

本版后台配置模块划分为 5 个：

1. 岗位管理
2. 模块中心
3. 功能授权
4. 字段标题中心
5. 状态权限中心

说明：

- 岗位管理负责岗位树与岗位员工。
- 功能授权消费岗位管理产生的岗位主数据。
- 模块中心负责模块元数据。
- 字段标题中心负责字段显示文案。
- 状态权限中心负责状态维度限制。

## 5. 页面设计

### 5.1 岗位管理

#### 页面目标

统一维护组织岗位树、岗位信息、岗位员工关系。

#### 页面布局

- 左侧：岗位树
  - 集团
  - 总公司
  - 分公司
  - 岗位
- 右侧：岗位工作区
  - 当前岗位摘要
  - 岗位员工预览
  - 添加员工
  - 查看员工

#### 左侧岗位树功能

- 搜索岗位名称或编码
- 新增岗位
- 新增下级岗位
- 编辑岗位

#### 右侧岗位工作区功能

- 展示岗位编码、所属组织、上级节点、岗位负责人
- 展示岗位员工数量、下级岗位数量、显示顺序
- 添加员工到当前岗位
- 查看当前岗位员工列表

#### 页面交互说明

1. 先在左侧岗位树选择岗位。
2. 左侧负责岗位树维护。
3. 右侧只对当前岗位做员工管理与信息查看。
4. 功能授权页面只读取岗位管理结果，不承担岗位主数据修改。

### 5.2 模块中心

#### 页面目标

维护模块元数据，形成运行时上下文的结构定义。

#### 页面布局

- 左侧：模块树
- 右侧：模块概览 + 4 个页签

#### 管理内容

- 模块编码、模块名称、模块类型
- 路由地址、组件路径、导航可见
- 状态型模块标记、状态字段
- 单头 Tab 与字段
- 明细 Tab 与字段
- 按钮定义与按钮区域

### 5.3 功能授权

#### 页面目标

按组织岗位配置模块的岗位基线权限。

#### 页面布局

- 左侧：岗位树
- 右侧顶部：当前岗位摘要
  - 当前岗位
  - 当前组织
  - 保存 Mock / 保存草稿
  - 恢复当前模块
  - 预览岗位基线
- 右侧主体：
  - 左栏模块树
  - 右栏权限编辑区

#### 权限编辑区页签

1. 模块授权
2. 按钮权限
3. 字段权限

#### 模块授权页签

配置：

- 模块是否可见
- 导航是否可见

#### 按钮权限页签

按按钮区域分组展示：

- `LIST_TOOLBAR`
- `LIST_ROW_BUTTON`
- `HEADER_TOOLBAR`
- `DETAIL_ROW_BUTTON`

每个按钮配置：

- 按钮编码
- 按钮标题
- 权限级别：不可见 / 禁用 / 可用

#### 字段权限页签

字段权限按前端页面结构管理：

- 先分 `单头字段 / 明细字段`
- 再按 Tab 分组展示

每个字段配置：

- 字段编码
- 字段标题
- 数据路径
- 权限级别：不可见 / 只读 / 可写

#### 页面边界

- 功能授权只负责岗位基线。
- 功能授权不编辑岗位主数据。
- 功能授权不选择状态。
- 状态权限一律在状态权限中心维护。

### 5.4 字段标题中心

#### 页面目标

维护模块字段的显示标题、占位提示、帮助说明。

#### 页面布局

- 左侧：模块树
- 右侧：单头字段 / 明细字段
- 再按 Tab 分组维护字段展示信息

#### 数据边界

字段标题属于模块元数据，不属于岗位权限，也不属于状态权限。

### 5.5 状态权限中心

#### 页面目标

在岗位基线之上维护状态附加限制。

#### 页面布局

- 左侧：仅展示状态型模块
- 右侧：状态定义、流转规则、按钮权限、字段权限

#### 配置规则

- 状态权限只做收紧
- 不允许通过状态配置把岗位不可见的资源改成可见
- 不允许通过状态配置把岗位只读字段改成可写

## 7. 表结构设计

### 7.1 组织与岗位

| 表名 | 说明 | 关键字段 |
| --- | --- | --- |
| `sys_org_unit` | 组织树 | `id`, `parent_id`, `org_code`, `org_name`, `org_type`, `sort`, `status` |
| `sys_org_post` | 组织岗位，支持岗位树 | `id`, `org_unit_id`, `parent_post_id`, `post_code`, `post_name`, `post_type`, `manager_user_id`, `sort`, `status`, `remark` |
| `sys_user_org_post` | 用户与组织岗位关系 | `id`, `user_id`, `org_post_id`, `is_primary`, `status` |

约束建议：

- `sys_org_unit.uk_org_code(org_code)`
- `sys_org_post.uk_org_post(org_unit_id, post_code)`
- `sys_user_org_post.uk_user_org_post(user_id, org_post_id)`

说明：

- `parent_post_id` 用于支撑岗位树。
- 顶层岗位 `parent_post_id` 为空或 `0`，表示直接挂在组织下。

### 7.2 模块元数据

| 表名 | 说明 | 关键字段 |
| --- | --- | --- |
| `sys_module` | 模块主表 | `id`, `parent_id`, `module_code`, `module_name`, `module_type`, `route_path`, `component_path`, `nav_visible`, `stateful_flag`, `state_field_code`, `context_version`, `status` |
| `sys_module_tab` | Tab 定义 | `id`, `module_id`, `tab_scope`, `tab_code`, `tab_name`, `sort`, `status` |
| `sys_module_field` | 字段定义 | `id`, `module_id`, `tab_id`, `field_scope`, `field_code`, `data_path`, `default_title`, `display_title`, `placeholder`, `component_type`, `value_type`, `required_flag`, `sort`, `status` |
| `sys_module_button` | 按钮定义 | `id`, `module_id`, `button_code`, `button_name`, `button_area`, `sort`, `status` |
| `sys_module_state` | 状态定义 | `id`, `module_id`, `state_code`, `state_name`, `sort`, `is_initial`, `is_final`, `status` |
| `sys_module_transition` | 状态流转 | `id`, `module_id`, `action_button_code`, `from_state_code`, `to_state_code`, `status` |

说明：

- `tab_scope` 取值：`HEADER`、`DETAIL`
- `field_scope` 取值：`HEADER`、`DETAIL`
- `button_area` 取值：`LIST_TOOLBAR`、`LIST_ROW_BUTTON`、`HEADER_TOOLBAR`、`DETAIL_ROW_BUTTON`
- 字段标题中心直接维护 `sys_module_field.display_title`、`placeholder`

### 7.3 岗位基线权限

## 9. 服务层建议

建议拆出以下核心服务：

- `PostManageService`
- `ModuleMetaService`
- `FunctionAuthService`
- `StateAuthService`
- `ModuleContextService`
- `ModuleActionGuardService`

职责边界：

- `PostManageService` 负责岗位树与岗位员工关系
- `FunctionAuthService` 只处理岗位基线权限
- `StateAuthService` 只处理状态附加权限
- `ModuleContextService` 统一组装前端运行时上下文
- `ModuleActionGuardService` 统一校验写操作

## 10. 代码约束

为彻底去掉冗余权限判断代码，强制约束如下：

1. Controller 不直接判断字段权限。
2. Controller 不直接判断按钮权限。
3. Service 不散落写 `if (post)`、`if (state)`、`if (field)` 分支。
4. 页面渲染只消费模块上下文，不自行推导权限。
5. 写操作统一经过 `ModuleActionGuardService`。

## 11. 本版替换关系

本版文档替换掉以下旧结论：

- “岗位直接融入功能授权页面并在该页编辑”
- “岗位页面不再独立存在”
- “功能授权页面可以切换状态进行授权”
- “基于 `bill_template` 的单据权限建模”
- “角色作为单据内部权限主体”

最终落地链路以此为准：

`岗位管理 -> 功能授权基线 -> 状态权限收紧 -> 模块上下文 -> 前端渲染 / 写操作校验`
