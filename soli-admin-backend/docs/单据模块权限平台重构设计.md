# 单据模块权限平台重构设计

## 1. 文档说明

本文档以当前仓库内已经落地的实现为准，覆盖以下范围：

- 模块中心
- 功能授权
- 字段标题中心
- 状态权限中心
- 初始化建表与种子数据

不再沿用旧版文档中的 `/admin/*` 接口示例，也不再描述前端 mock 方案。当前前端页面已经全部切换为后端数据驱动。

## 2. 当前实现结论

### 2.1 平台拆分

当前后台管理端围绕“模块元数据 + 岗位基线权限 + 状态收紧权限”拆分为 4 个核心页面：

1. 模块中心
2. 功能授权
3. 字段标题中心
4. 状态权限中心

岗位主数据仍由现有岗位管理能力承载，权限页只消费岗位树和岗位详情，不负责编辑岗位主数据。

### 2.2 权限模型

当前实现的权限边界如下：

- 模块中心维护模块、Tab、字段、按钮、状态、流转等元数据
- 字段标题中心维护字段展示元数据：`displayTitle`、`placeholder`、`helpText`
- 功能授权维护岗位基线权限
- 状态权限中心只维护状态维度的附加限制，只允许“收紧”，不允许“放大”

统一权限级别如下：

- 字段权限：`0 = 隐藏`，`1 = 只读`，`2 = 可写`
- 按钮权限：`0 = 隐藏`，`1 = 禁用`，`2 = 可用`
- 状态字段限制：`0 = 隐藏`，`1 = 只读`，`2 = 不收紧`
- 状态按钮限制：`0 = 隐藏`，`1 = 禁用`，`2 = 不收紧`

最终运行时权限计算规则仍为：

- `finalFieldLevel = min(orgPostFieldLevel, stateFieldLimitLevel)`
- `finalButtonLevel = min(orgPostButtonLevel, stateButtonLimitLevel)`

当状态权限表中不存在记录时，默认按 `2 = 不收紧` 处理，即直接继承岗位基线结果。

### 2.3 状态模型

当前种子数据内置了 5 个标准状态：

- `unaudited`
- `pre_audited`
- `audited`
- `shipped`
- `completed`

`create` 不再作为独立状态处理，初始单据统一归入 `unaudited`。

## 3. 页面实现说明

### 3.1 模块中心

模块中心当前职责：

- 维护模块树
- 维护模块基础信息
- 维护 Tab、字段、按钮
- 维护状态与流转定义
- 提供上下文预览

当前模块详情接口返回的数据已经包含：

- `headerTabs`
- `detailTabs`
- `buttons`
- `states`
- `transitions`

字段定义中已包含以下可展示元数据：

- `defaultTitle`
- `displayTitle`
- `placeholder`
- `helpText`

说明：

- `GET /sys/module/{id}/context-preview` 当前仅用于模块中心做结构预览
- 该接口不再注入 mock 的当前岗位或当前状态
- 当前返回的是模块结构上下文，不是完整运行时权限上下文

### 3.2 功能授权

功能授权页当前全部使用后端数据：

- 左侧岗位树来自后端岗位树接口
- 中间模块树来自后端模块树接口
- 右侧配置区通过 `orgPostId + moduleId` 查询岗位基线权限
- 保存时回写岗位模块权限、字段权限、按钮权限

前端不再依赖本地 mock 生成岗位权限数据。

### 3.3 字段标题中心

字段标题中心当前全部使用后端数据：

- 模块树来自后端
- 模块详情直接复用模块中心详情数据
- 保存时批量更新 `sys_module_field.display_title / placeholder / help_text`

前端不再维护本地标题 mock 仓库。

### 3.4 状态权限中心

状态权限中心当前全部使用后端数据：

- 左侧只展示 `stateful_flag = 1` 的模块树
- 页面详情直接返回模块详情 + 当前模块状态限制配置
- 保存时批量覆盖当前模块的状态字段限制、状态按钮限制

实现约束：

- 只保存“收紧”结果
- `permission_level >= 2` 的状态限制不会落库
- 因此数据库里只保存状态下真正被收紧的资源

## 4. 数据库设计

### 4.1 已落地表

当前 `sql/init_Table.sql` 已包含以下与本次重构直接相关的表：

#### 模块元数据

- `sys_module`
- `sys_module_tab`
- `sys_module_field`

- `sys_module_state`

#### 岗位基线权限

#### 状态附加限制

- `sys_module_state_field_auth`
- `sys_module_state_button_auth`

#### 其他

- `sys_module_publish_log`

### 4.2 关键字段

#### `sys_module_field`

当前已支持以下展示类字段：

- `default_title`
- `display_title`
- `placeholder`
- `help_text`

#### `sys_module_state`

用于维护模块状态定义：

- `state_code`
- `state_name`
- `is_initial`
- `is_final`
- `sort`
- `status`

#### `sys_module_transition`

用于维护状态流转：

- `action_button_code`
- `from_state_code`
- `to_state_code`
- `sort`
- `status`

- `sys_org_post_module_auth` 保存模块是否可见、导航是否可见
- `sys_org_post_field_auth` 保存岗位字段基线权限
- `sys_org_post_button_auth` 保存岗位按钮基线权限

- `sys_module_state_field_auth` 保存状态字段收紧规则
- `sys_module_state_button_auth` 保存状态按钮收紧规则

### 5.3 admin 权限处理

当前实现已经把模块权限平台相关权限放给 admin 用户，具体包括两层：

#### 菜单 / 接口权限

`init_Data.sql` 已插入以下权限点并授予 admin 角色：

- `sys:module:page`
- `sys:module:tab:modify`
- `sys:module:field:modify`
- `sys:module:button:modify`
- `sys:function-auth:page`
- `sys:function-auth:save`
- `sys:module-title:page`
- `sys:module-title:save`
- `sys:state-auth:page`
- `sys:state-auth:save`

#### 岗位基线权限

当前初始化逻辑中：

- admin 用户通过 `sys_user_org_post` 绑定到岗位 `9101`
- `9101` 已被赋予采购、销售两个目录及两个单据模块的可见权限
- `9101` 已被赋予所有初始化字段的 `2 = 可写`
- `9101` 已被赋予所有初始化按钮的 `2 = 可用`

说明：

- 当前没有为状态权限表预置收紧数据
- 这样 admin 默认不会被状态权限额外限制

## 7. 前端对接方式

当前前端已移除模块权限平台相关的 mock 文件，页面全部改为调用真实接口。

对应 API 文件如下：

- `soli-admin-frontend/src/api/moduleCenter.ts`
- `soli-admin-frontend/src/api/functionAuth.ts`
- `soli-admin-frontend/src/api/moduleTitle.ts`
- `soli-admin-frontend/src/api/stateAuth.ts`

页面数据装配方式如下：

- 功能授权页把后端返回的 `fieldId/buttonId` 权限列表映射为前端按 `fieldCode/buttonCode` 管理的表单结构
- 状态权限页把后端返回的 `permissionsByState` 映射为按 `stateCode -> fieldCode/buttonCode` 管理的编辑结构
- 字段标题中心直接基于模块详情中的字段定义编辑 `displayTitle/placeholder/helpText`

说明：

- 前端页面不再从本地 fixture 组装模块、岗位、状态权限
- 页面中的预览抽屉也是基于后端数据即时组装，不再读取 mock 上下文

## 8. 当前实现边界

以下内容当前未做成完整运行时能力，文档中需要明确区分：

### 8.1 已实现

- 模块中心元数据维护
- 岗位基线权限维护
- 字段标题覆盖维护
- 状态收紧权限维护

### 8.2 未实现为统一运行时接口

当前还没有落地“按用户当前岗位 + 单据当前状态 + 字段标题覆盖”统一合并后的运行时接口。

现状是：

- 模块中心有 `GET /sys/module/{id}/context-preview`
- 该接口只做模块结构预览
- 不负责拼装真实岗位权限
- 不负责拼装真实状态限制
- 不负责根据业务单据主键推导当前状态

如果后续要接入真实运行时渲染，需要在此基础上新增独立的运行时上下文服务。

## 9. 本次重构后的约束

后续继续开发时，按以下原则执行：

1. 模块结构只在模块中心维护。
2. 字段展示文案只在字段标题中心维护。
3. 岗位基线权限只在功能授权维护。
4. 状态附加限制只在状态权限中心维护。
5. 状态权限只能收紧，不能放大岗位基线。
6. 前端页面不得再回退到本地 mock 数据。
