# Soli Admin 项目介绍

## 1. 项目定位

Soli Admin 是一个面向企业内部管理系统的前后端分离后台平台，适合采购、组织、人事、主数据、配置中心等场景快速落地。

它不是单纯的“菜单 + 按钮权限”后台，而是把模块、字段、按钮、状态、字段标题、岗位权限全部统一建模，做到：

- 页面可以配置
- 权限可以细到字段和按钮
- 状态流转可以驱动界面行为
- 前端不需要自己拼装权限数据
- 业务模块可以按统一标准接入

当前仓库已经落地了采购订单、用户管理、岗位管理、字典管理、参数管理、功能授权、字段标题、状态权限、模块管理等能力。

## 2. 适合谁使用

这个项目尤其适合下面几类团队：

- 正在做 ERP、SCM、采购、供应链、财务协同等后台系统的团队
- 需要做“不同岗位看到不同字段、不同状态允许不同操作”的团队
- 需要多公司、多组织、多岗位权限模型的团队
- 想要一个能持续扩展业务模块，而不是每次都从零搭页面和权限的团队
- 想把前端页面配置、字段显示、按钮权限统一沉淀的平台型团队

## 3. 核心价值

### 3.1 模块即配置

系统把一个业务页面拆成模块、组件、字段、按钮、状态几层结构。新增一个业务模块，不只是新增页面，还能同步建立：

- 页面基础信息
- 组件结构
- 字段定义
- 字段标题
- 按钮定义
- 状态流转
- 岗位权限
- 状态权限

这使得系统天然适合做“可持续扩展的业务后台”。

### 3.2 字段标题独立可控

字段标题不是前端写死，也不是运行时临时拼接，而是单独落表维护。字段标题表保留了运行时真正需要的标题、占位提示、帮助文本、组件类型、数据路径等信息，可以支持：

- 用户自定义显示标题
- 多语言扩展
- 表单提示文案个性化
- 运行时不再依赖模块字段表联查

### 3.3 权限不只到菜单

项目把权限拆成两层：

- 岗位维度权限
- 状态维度权限

运行时最终权限是两者叠加后的结果，所以一个按钮不是“有没有菜单权限”这么简单，而是要同时满足：

- 当前用户所属岗位是否允许
- 当前单据状态是否允许

这非常适合审批流、单据流、业务流场景。

### 3.4 前后端职责清晰

后端统一返回模块上下文，前端只消费结果。这样做有几个直接好处：

- 前端不需要自己组装字段权限
- 前端不需要自己推导按钮权限
- 同一个模块在多个页面形态下可以保持一致行为
- 权限逻辑统一收口在服务端

### 3.5 采购订单是可复用业务样板

采购订单模块已经把一个真实业务单据页面所需的大部分能力跑通了：

- 单据分页
- 单据详情
- 明细行
- 来源单据
- 附件
- 操作日志
- 状态流转
- 按钮权限控制
- 模块上下文对接

如果团队后续要扩展销售订单、入库单、出库单、请购单，基本可以沿用这套模式。

## 5. 和传统后台相比，这个项目的优势

传统后台大多数只能做到：

- 页面写死
- 菜单权限
- 按钮权限

Soli Admin 进一步往前走了一步，形成了“模块平台 + 业务模块”的结构，优势在于：

- 可以做真正的平台化二开
- 可以支撑复杂表单和复杂单据
- 可以把字段控制权交给业务配置
- 可以在同一页面里按状态变化权限
- 可以让一个模块的前端和后端围绕统一上下文协作

## 6. 典型应用场景

- 企业采购管理系统
- 内部协同平台
- 供应链中台后台
- 制造业业务单据平台
- SaaS 后台的租户化管理端
- 需要精细权限的中后台系统

## 7. 为什么值得尝试

如果你的团队已经遇到下面这些问题，这个项目会很合适：

- 每新增一个页面都要重复写表单、权限、按钮逻辑
- 字段标题和页面文案散落在前端，难统一维护
- 不同岗位、不同状态下界面行为不一致，容易出错
- 采购、审批、单据类业务越来越多，但平台能力没有沉淀

Soli Admin 的价值就在于，它已经把这些“重复搭地基”的工作提前做好了。你可以在这个基础上，直接开始构建自己的业务模块，而不是每个项目重新造一套后台框架。

## 8. 快速体验建议

建议按下面顺序体验项目：

1. 导入 [init_Table.sql](../soli-admin-backend/sql/init_Table.sql) 和 [init_Data.sql](../soli-admin-backend/sql/init_Data.sql)
2. 启动后端聚合应用 `soli-admin-start`
3. 启动前端项目 `soli-admin-frontend`
4. 使用初始化账号体验登录、切换公司、查看导航
5. 重点体验采购订单、用户管理、岗位管理、字段标题、状态权限几个模块
6. 观察模块上下文如何驱动字段显示、编辑能力和按钮可用性

## 9. 总结

Soli Admin 不是一个只适合演示的后台模板，而是一套已经具备平台思路的业务后台底座。

如果你要做的是：

- 复杂单据系统
- 多岗位权限系统
- 可配置字段系统
- 持续演进的企业后台

那么这个项目值得直接拿来作为基础版本继续建设。
